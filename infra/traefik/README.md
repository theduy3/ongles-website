# infra/traefik

Hand-managed Traefik dynamic config for the VPS (`147.93.116.94`, Dokploy + Traefik
file provider at `/etc/dokploy/traefik/dynamic/`).

These files are **not** generated by Dokploy. Dokploy owns the per-app
`<app>-<hash>.yml` files and regenerates them on every redeploy — so anything
hand-edited there is clobbered. Config that must survive redeploys lives here and
is copied to the VPS manually.

## `ongles-apex-redirect.yml`

Apex → www `301` redirects + LE certs for the three ongles tenants
(`onglesmaily.com`, `onglescharlesbourg.com`, `onglesrivieres.com`).

Dokploy only registers the `www.` host per tenant, leaving the bare apex with no
router (TLS handshake fails). This file adds, per tenant:

- a `web` (:80) apex router → `ongles-redirect-to-www` middleware
- a `websecure` (:443) apex router → same middleware, `tls.certResolver: letsencrypt`
  so the apex gets its own cert
- a shared `redirectRegex` middleware rewriting `https?://(www.)?<apex>/<path>` →
  `https://www.<apex>/<path>` (permanent)
- a dummy unreachable service (`127.0.0.1:9`) — the redirect short-circuits before
  it is ever dialed

ACME's HTTP-01 challenge handler outranks the redirect on
`/.well-known/acme-challenge/*`, so apex cert issuance still works.

### Deploy / update

```bash
scp infra/traefik/ongles-apex-redirect.yml \
  paperclip:/etc/dokploy/traefik/dynamic/ongles-apex-redirect.yml
# Traefik file provider hot-reloads (watch: true) — no restart needed.
```

Verify:

```bash
for h in onglesmaily.com onglescharlesbourg.com onglesrivieres.com; do
  curl -s -o /dev/null -w "$h %{http_code} -> %{redirect_url}\n" "https://$h/"
done   # expect: 301 -> https://www.<h>/
```
